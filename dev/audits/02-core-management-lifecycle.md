# Core Management & Lifecycle — Deep Audit

## Summary

Overall grade: **C**. The core-management slice is architecturally serious — it documents a 4-level lock hierarchy, uses `SafeBinaryMutex`, RAII lock guards, a render-thread-deferred device-destroy path, and DEV-only lock-order guards. That is unusually disciplined for Godot module code. But the implementation has a pile of concrete hazards that undermine the discipline: the module shutdown order is wrong and can free a `RenderingDevice` before the scene director's shared renderer has given back its RIDs; `GaussianSplattingPerformanceMonitors::create_singleton()` is called from two places with no synchronization; `GaussianSplatSceneDirector::~D` holds no mutex when wiping `worlds`; the monitors singleton holds naked pointers to `TileRenderer`/`GaussianSplatRenderer` with no protection against destructor races or against concurrent `Performance` callbacks; `get_primary_rendering_device()` silently bypasses the whole local-device machinery and returns the main RD, which makes half of the rest of the class dead code with an attractive shape. Under load this will look fine until a race lands, at which point triage is brutal because ownership is spread across five singletons.

## What this code does

- **`GaussianSplatManager`** — engine-level singleton that owns GPU buffer registries (`gaussian_buffers`, `dynamic_asset_cache`), registers all `rendering/gaussian_splatting/*` project settings, tracks active `GaussianSplatNode3D`s, and manages lazy creation/teardown of two `RenderingDevice` instances (primary + shared submission). Exposes `ScopedSubmissionLock` for callers that need a device + mutex.
- **`GaussianSplatSceneDirector`** — world/scenario-indexed registry of `SharedWorld { scenario, renderer, instances, asset_records, world_submission }`. Owns the per-world shared `Ref<GaussianSplatRenderer>`, arbitrates world-submission contracts with rollback, computes instance LOD, and builds `InstanceDataGPU` buffers.
- **`GaussianSplatConfigRegistry`** — static init-order list for the seven config subsystems (sort/SH/F16/quant/LOD/pipeline/settings).
- **`GaussianSplatSettingsManager`** — thin wrapper around project settings for four debug-overlay bools, with editor-only persistence.
- **`GaussianSplatMergeUtils`** — functional asset-merge: concatenates N `GaussianSplatAsset`s into one `GaussianData` + spatial chunk list.
- **`QualityTierConfig`** — lookup-table resolver from preset string ("low", "rtx_3090_4k", "steam_deck", ...) to a struct of caps/toggles.
- **`ResidencyBudgetController`** — pure-function gate for streaming: given loaded-chunk count + frame budget + policy, decide LoadDirect / EvictThenLoad / Skip.
- **`GaussianSplattingPerformanceMonitors`** — `Object` singleton that registers ~100 custom Godot Performance monitors and drives them from two active-renderer pointers.
- **`PainterlyManager`** — deterministic per-splat painterly metadata (seed, blue noise, jitter) keyed by base seed × splat index × LOD level.

## Strengths

- Explicit, written-down lock hierarchy with four levels and a DEV-only `_GSLockLevelGuard` that trips on out-of-order acquisition (`gaussian_splat_manager.h:54-72`, `gaussian_splat_manager.cpp:56-91`). This is already better than 99% of Godot module code.
- `ScopedSubmissionLock` is a real RAII guard: non-copyable, move-correct, resets the lock-level tracker on release (`gaussian_splat_manager.cpp:164-214`). The move-assignment `_release()`-then-steal pattern is correct.
- Device teardown is render-thread-deferred through `RenderThreadDispatcher` with a two-mutex hand-off (request serialization + pending-pointers mutex) and an explicit timeout fall-through path (`gaussian_splat_manager.cpp:361-415`). The author clearly thought about "what if the render thread has already died".
- `ResidencyBudgetController` ships a runtime invariant checker (`_admission_gate_invariant_holds`) that double-checks the decision function and forces `Skip` on violation (`residency_budget_controller.cpp:7-54`). The same file rejects eviction-budget underflow via `ERR_FAIL_COND_MSG`.
- `_release_registered_resources()` (`gaussian_splat_manager.cpp:287-359`) correctly preserves entries whose owner device is missing rather than `free()`-ing on a wrong device, and logs the deferred-release count. That is unusual defensive discipline.
- RenderDoc-compatibility mode is a real pragma: env-probe + DLL probe + project-setting force-switch, and every `_request_*_local_device` / `_create_local_device_on_render_thread` short-circuits (`gaussian_splat_manager.cpp:105-129`, `481-546`).

## Top issues

- **[severity: crash]** `modules/gaussian_splatting/register_types.cpp:201-214` — Shutdown order is `GaussianSplatManager` first (including `memdelete` → `_destroy_local_devices()`), then `GaussianSplatSceneDirector`. The director's dtor calls `worlds.clear()` (`gaussian_splat_scene_director.cpp:190-198`) which drops every `Ref<GaussianSplatRenderer>`. Those renderers own RIDs allocated against `GaussianSplatManager::get_primary_rendering_device()` and will call `RenderingDevice::free(...)` from inside `~GaussianSplatRenderer`. By that point the manager is gone — no local device, no singleton — but `RS::get_rendering_device()` (the main device) may still exist, so this often hides. **Why it matters**: the instant the renderer is changed to use a local device, or a shutdown happens during a hot-reload where `RS` is torn down before modules, this goes from "works by luck" to UAF or double-free on every exit. **Fix direction**: destroy director before manager, or have manager's dtor pump director's `worlds.clear()` first as a pre-step. Assert invariant.
- **[severity: crash]** `performance_monitors.cpp:90-95` + `renderer/gaussian_splat_renderer.cpp:1022` — `GaussianSplattingPerformanceMonitors::create_singleton()` is called from both `register_types.cpp:121` (module init) and from a renderer constructor. The check `if (!singleton)` is not atomic. Two `GaussianSplatRenderer`s being constructed off the render thread while the module finishes init (plausible during editor hot-reload or headless tests) race the pointer. **Why it matters**: non-atomic write to `singleton` + `initialize_monitors()` running twice against `Performance`, registering monitors twice and fighting `remove_custom_monitor`. **Fix direction**: make `create_singleton()` idempotent + atomic (CAS), or forbid the renderer path entirely and require register_types to own it.
- **[severity: corruption]** `performance_monitors.cpp:104-154` — `register_renderer`, `unregister_renderer`, `register_splat_renderer`, `unregister_splat_renderer` mutate `registered_renderers` and `active_renderer`/`active_splat_renderer` with no synchronization, but the header comment at `performance_monitors.h:27-31` only promises "main thread" safety. The monitor callbacks (`_get_*`) are invoked from whichever thread `Performance` decides to poll on; the cached `_streaming_snapshot_for_monitors` uses `thread_local` precisely because "callback threading changes". So the author knows the callbacks can be off-thread, yet every `active_splat_renderer` read is unguarded. **Why it matters**: `unregister_splat_renderer` can null out `active_splat_renderer` between the null-check and the method call in e.g. `_get_gpu_frame_time_ms` (`performance_monitors.cpp:516-523`). Raw pointer, no refcount. **Fix direction**: hold a `Mutex` around registration and snapshot capture, or convert to `Ref<>`/`ObjectID`-indirect with `ObjectDB::get_instance()`.
- **[severity: crash]** `gaussian_splat_scene_director.cpp:184-198` — Constructor and destructor touch the `singleton` pointer and the `worlds` map with **no mutex**. Director methods (`register_instance`, `update_instance_lods`, `build_instance_buffer_for_renderer`, all 1000+ LOC of them) do hold `world_mutex`. If a render thread holding `world_mutex` is in the middle of `build_instance_buffer_for_renderer` while the main thread enters `~GaussianSplatSceneDirector()` → `worlds.clear()` (which does not take the mutex), you get concurrent mutation of a HashMap being iterated. **Why it matters**: shutdown is when race windows open widest. **Fix direction**: take `world_mutex` in the dtor; acquire/release before clearing. Same for ctor's `singleton=this` write — document or guard.
- **[severity: crash]** `gaussian_splat_manager.cpp:657-671` — `~GaussianSplatManager()` acquires `active_nodes_mutex` (level 3) then `_release_registered_resources()` (level 2) then `_destroy_local_devices()` (level 4). The comment claims this is sequential, not nested, which is true only because level-3 scope closes before level-2 opens. But the code does not reset `_gs_held_lock_level` between the `MutexLock` destructor and the next `GS_LOCK_ORDER_GUARD` — only because the guard struct's dtor fires, which it does. This works today, but the dtor acquiring a level-3 lock before any level-2 operation technically violates the documented ordering ("outermost to innermost"). **Why it matters**: any future tweak that adds a second lock inside the same scope will silently go wrong. **Fix direction**: reorder to level 2 → 3 → 4, or split the dtor into a `shutdown()` that is explicit.
- **[severity: corruption]** `gaussian_splat_manager.cpp:757-799` — `unregister_gaussian_buffer` decrements `entry->ref_count` **after** it has captured the `owner_device` into a local. If decrement leaves ref at zero but `owner_device` was `nullptr` (the `entry->ref_count == 1` path only populates `owner_device` when it's 1, but a caller that hits the `--entry->ref_count > 0` path and somehow left `owner_device` unset... actually let me be precise: the code at line 780 guards `owner_device` acquisition by `entry->ref_count == 1 && entry->gpu_buffer.is_valid()`. Then line 787 decrements; if the new value is 0, line 791 dereferences `owner_device`. The `if (entry->ref_count == 1 && ...)` predicate **before** the decrement is correct, but it fires `ERR_FAIL_NULL_MSG` at 784 which aborts the function. Good. But note `owner_device` is still used when `ref_count` transitions from 1→0, and the `ERR_FAIL_NULL_MSG` emits an error **but the invariant restored**: no free. Then `entry` still has `ref_count == 1` (the ERR_FAIL returned before decrement). Next caller also ERR_FAILs. The buffer leaks but doesn't crash. Acceptable but silent. **Why it matters**: unbounded refcount drift if owner metadata is dropped. **Fix direction**: erase the broken entry, log loudly, and forget the buffer rather than permanently failing every future unregister.
- **[severity: crash]** `gaussian_splat_manager.cpp:1164-1185` — `get_primary_rendering_device()` returns `RenderingServer::get_singleton()->get_rendering_device()` (the main RD) **before** consulting `primary_local_device`. The entire `_request_primary_local_device()` / `_create_local_device_on_render_thread` machinery is dead code on any non-RenderDoc run. Comments on 1164-1169 say this is the "CRITICAL FIX". **Why it matters**: the class documents "local RenderingDevice instances bound on the render thread" (header line 115-117) and the whole `ScopedSubmissionLock` pretends to hand out the shared local device via `_ensure_shared_submission_device_locked()` (1208-1219). A caller that assumes `ScopedSubmissionLock::get_rendering_device()` returns the shared local device (distinct from main) will do cross-device work; the caller gets the main device sometimes and the local device other times. The fallback at line 1200 is exactly this: if shared device is disabled, return main RD — now all "submission-locked" work silently runs on the main RD with no mutex. **Fix direction**: delete local-device creation entirely, or commit to it. The current hybrid is worse than either.
- **[severity: perf]** `gaussian_splat_scene_director.cpp:271-296, 317-338` — `_find_world_for_instance`, `_find_world_for_renderer`, and `_find_world_for_world_submission` all linear-scan `worlds` under `world_mutex` on every `update_instance_transform` / `update_instance_params` / `unregister_instance` / `update_instance_lods_for_renderer` / `get_instance_submission` / `has_world_submission_for_renderer` / `submit_world_submission`. With N worlds and M instances, instance bookkeeping is O(N) per call. The `instance_lookup: HashMap<ObjectID, uint32_t>` exists per-world and is O(1); the top-level map is by RID scenario. **Why it matters**: editor with three viewports = three scenarios; per-frame the director does `O(worlds)` work per instance even though a reverse index `ObjectID → scenario` would be O(1). **Fix direction**: keep a `HashMap<ObjectID, RID>` alongside, populated by `register_instance`, consulted by the updaters; the linear scan stays only as a reconciliation fallback.
- **[severity: corruption]** `gaussian_splat_scene_director.cpp:816-821` — `unregister_instance` does the swap-and-pop: `world->instances[index] = world->instances[last_index]; world->instance_lookup[...new_index...] = index; world->instances.remove_at(last_index);`. If `index == last_index`, the self-assignment line is redundant but harmless. However, `world->instance_lookup.erase(p_node_id)` happens at line 822 **after** `world->instance_lookup[world->instances[index].node_id] = index;` — which, when `index == last_index`, overwrites the same key that is about to be erased. The code currently guards with `if (index != last_index)`, so the only failure mode is if `instances[index]` post-swap is the removed node — but `remove_at(last_index)` happens after, so it is safe. Fine in practice but the ordering is load-bearing and unobvious. **Why it matters**: one wrong edit and this becomes a stale-index UAF. **Fix direction**: extract a `_swap_remove_instance()` helper with the invariants documented.
- **[severity: maint]** `gaussian_splat_scene_director.cpp:13-25, 898` — `_is_scene_director_log_enabled()` re-reads three `ProjectSettings` entries **every frame**, called from `update_instance_lods` (and its per-renderer variant) and `build_instance_buffer_for_renderer`. `ProjectSettings::get_setting` takes a lock and allocates a `String`. **Why it matters**: under heavy debug-log off, this is still three string-keyed hashmap lookups + lock acquisitions per frame per world, per build/LOD call. **Fix direction**: cache in the director, invalidate on a known signal, or gate by a single `SafeFlag` updated via a project-setting watcher.
- **[severity: maint]** `gaussian_splat_manager.cpp:971-1151` — `initialize_module()` registers ~100 project settings inline. This function is untestable, un-searchable, and mirrors scattered reads in the ctor (274-284) that re-read a subset of the same settings before `GLOBAL_DEF` has even registered defaults for them. **Why it matters**: the order is "read from project file in ctor at the moment the class is first instantiated, then register defaults in `initialize_module`". Tests that instantiate a `GaussianSplatManager` directly (there are at least five: `test_config_validation.h:33`, `test_gaussian_splatting.cpp:29`, `test_gpu_streaming.cpp:2937/3114/3218`, `test_gpu_streaming.h:329`, `test_integration.cpp:239`) get whatever the registered defaults from a **prior** module init left behind, or nothing. Non-deterministic test setup. **Fix direction**: split settings registration into a pure `register_project_settings()` that runs first in module init, before any ctor.
- **[severity: perf]** `performance_monitors.cpp:57-77` — The monitor snapshot cache is `thread_local`, so every thread that polls the `Performance` singleton materializes its own `MonitorStreamingSnapshot` (hundreds of bytes) and keeps it alive for process lifetime. The comment acknowledges "expected on the main thread, but keep cache state thread-local to avoid cross-thread mutation if callback threading changes". **Why it matters**: if `Performance` ever polls from a worker pool, N workers × snapshot-size of permanently-live memory per renderer. And the cache doesn't key on active-renderer identity, so a renderer swap leaves stale cached data until the next frame. `cached_key == cache_key` with `cached_renderer == p_renderer` handles that, but `_monitor_snapshot_cache_key` builds a 64-bit hash that can collide; nothing in the callback path tolerates a stale snapshot. **Fix direction**: single cache behind a `Mutex`, keyed on `(renderer, frame_counter)`, invalidated on `unregister_splat_renderer`.
- **[severity: maint]** `gaussian_splat_manager.h:73, 104-108` — The singleton is a plain `Object` with four `HashMap`s + two `Mutex`es inline, plus four atomic pointers. There is no separation between **engine-binding-facade** (the `GDCLASS` / `_bind_methods` / `ClassDB::bind_method` role) and **GPU resource registry** (the `register_gaussian_buffer` / `acquire_dynamic_asset` role). Consequence: every test file that needs any of the 20+ project settings has to `memnew(GaussianSplatManager)` and deal with RenderDoc probing, device requests, project-settings reading, and frame-callback plumbing. **Fix direction**: extract `GPUBufferRegistry` + `SubmissionDeviceManager` + `GSProjectSettings` as separate classes; keep `GaussianSplatManager` as a thin coordinator.

## Cross-cutting patterns

- **Double-singleton pattern everywhere** — `singleton = this` in ctor, `singleton = nullptr` in dtor, with no re-entrancy guard. `GaussianSplatManager` checks `ERR_FAIL_COND_MSG(singleton != nullptr, ...)` (line 217) and then assigns; `GaussianSplatSceneDirector` uses `if (!singleton)` (line 185), silently ignoring the second instance; `GaussianSplattingPerformanceMonitors` uses `if (!singleton)` in `create_singleton` (line 91) and has zero protection against concurrent creation. Three different idioms in four files for the same problem.
- **Linear scans of small collections masked as correctness** — `SceneDirector` linear-scans `worlds` for every lookup because "there are few worlds". Same pattern in `PerformanceMonitors::_get_active_splat_renderer` (loops all registered renderers to pick one with streaming data). Neither bounds the collection, so "small" is a runtime bet.
- **`thread_local` abuse** — `performance_monitors.cpp:60-62` uses `thread_local` caches to sidestep a threading question the author didn't want to resolve. This will come back. The project-memory even explicitly flags this as a past problem class: "Using wrong pattern (will crash in shipping builds)" appears in CLAUDE.md for a sibling UE5 project.
- **Lock hierarchy defined but not enforced outside the manager** — `SceneDirector::world_mutex` has no declared level. Is it above or below `resource_maps_mutex`? The director calls `GaussianSplatManager::get_singleton()->get_primary_rendering_device()` at line 219 while holding `world_mutex`. `get_primary_rendering_device()` may enter `_request_primary_local_device` which calls back into `RenderingServer::call_on_render_thread`, which can fire synchronously on the render thread and then deep into `_create_local_device_on_render_thread` — none of which re-enters `world_mutex` today but nothing in the contract forbids it.
- **Settings read paths diverge from write paths** — the settings manager persists only in editor (`TOOLS_ENABLED`), but the getters (`get_debug_show_*`) read unconditionally and always fall back to `false`. Runtime set never round-trips.
- **`ERR_FAIL_*` used as silent loss** — several hot paths (e.g. `register_gaussian_buffer` owner-mismatch at line 729, `unregister_gaussian_buffer` at line 777) treat invariant violations as "log + return", so a bug upstream becomes a silent stat drift rather than a test failure.

## Recommended refactor moves

- **P0 (must, ~2 days):** Fix shutdown order in `register_types.cpp:185-219` — destroy `GaussianSplatSceneDirector` before `GaussianSplatManager`, and assert that the scene director drained its renderers before the manager frees devices. Add a single integration test that runs init → create shared renderer → register an asset → shutdown, and verifies zero leaked RIDs and zero "missing owner device" warnings.
- **P0 (must, ~1 day):** Guard `GaussianSplattingPerformanceMonitors::create_singleton()` with a `std::once_flag` or CAS, and pick one call site. The renderer-constructor call path at `renderer/gaussian_splat_renderer.cpp:1022` should either be deleted or gated on "main thread + module initialized".
- **P0 (must, ~1 day):** Take `world_mutex` in `~GaussianSplatSceneDirector()` before `worlds.clear()`. Same-file, 3-line change. Same treatment in ctor if concurrent access to the singleton pointer is even theoretically possible.
- **P1 (should, ~3 days):** Add `ObjectID → RID` reverse index in `SceneDirector` to remove the O(worlds) scans. Same change removes the duplicated `_find_world_for_*` helpers.
- **P1 (should, ~2 days):** Resolve the local-device story. Either delete `primary_local_device` + `shared_submission_device` and all supporting machinery (saves ~200 LOC), or commit to it and make `get_primary_rendering_device()` actually return the local device. Document the contract.
- **P1 (should, ~1 day):** Replace `thread_local` monitor cache with a `Mutex`-guarded single cache keyed on `(renderer, frame_counter)`. Add an invalidation path in `unregister_splat_renderer`.
- **P1 (should, ~2 days):** Split `GaussianSplatManager::initialize_module()` into `register_project_settings()` (idempotent, runs first) + `bind_engine_hooks()` (runs after settings). Re-run tests that construct managers directly.
- **P2 (nice, ~1 week):** Extract `GPUBufferRegistry` from `GaussianSplatManager` as a standalone class with its own lock. `dynamic_asset_cache` and `gaussian_buffers` do not need to live next to project-settings boilerplate.
- **P2 (nice, ~2 days):** Add lock-level annotations to `SceneDirector::world_mutex` and `PerformanceMonitors` internal state. Run the DEV lock-order guard across the director too.
- **P2 (nice, ~3 days):** Replace `Performance`-callback raw-pointer design with `ObjectID`-indirect lookups so the monitors singleton cannot dereference dead renderers.

## Blind spots

- I did not open `renderer/gaussian_splat_renderer.cpp` beyond verifying the second `create_singleton()` call site at line 1022. The claim that `~GaussianSplatRenderer` frees RIDs against the manager's device is inferred from `SharedWorld::renderer` being a `Ref<GaussianSplatRenderer>` and from general Godot patterns; I did not confirm which device is stored and whether it's refcounted. If the renderer holds its own `Ref` to the device or uses a stable `RS::get_rendering_device()` call, the shutdown crash severity drops from crash to corruption-under-configuration.
- `RenderThreadDispatcher` (`interfaces/render_thread_dispatcher.h`) is outside my slice; I assumed `dispatch_call_on_render_thread_blocking` does not itself grab any of the four declared locks. If it does, the device-destroy path in `_dispatch_local_device_destroy_on_render_thread` (line 361-389) could deadlock against the submission mutex under contention.
- `SafeBinaryMutex<TAG>` semantics (`core/os/safe_binary_mutex.h`) — I assumed it is a plain exclusive non-recursive binary mutex with a per-thread TLS probe for debug. If recursion is allowed, my analysis of the dtor lock-order sequence is too strict.
- `Engine::Singleton` ownership contract after `remove_singleton` — I assumed `remove_singleton` does not call `memdelete` on the pointer, so `register_types.cpp:203-207` is correct in doing `remove → finalize → memdelete`. If Godot changed that, the explicit `memdelete` is a double-free.
- Tests (`tests/test_config_validation.h` etc.) were scanned for `memnew(GaussianSplatManager)` call-count only; I did not read the fixtures. It is possible they set `gaussian_splat_manager_singleton = nullptr` explicitly between tests.
- I did not run the DEV lock-ordering asserts under any stress workload; the claim "no method currently nests two of these locks in the same scope" in the header comment at line 64-65 was spot-checked by `grep`, not exhaustively proven.
- `GaussianSplatNode3D` (the registered node type in `active_nodes`) lives outside this slice, so I could not verify that `process_gaussian_render()` (called at `gaussian_splat_manager.cpp:636` from the deferred main-thread callback) is main-thread-safe.
- `painterly_manager.cpp:26-37` references `GaussianData::painterly` field and `Vector<GaussianData>::write[]`, which implies `GaussianData` is a value type in some compilation contexts despite being used as `Ref<GaussianData>` elsewhere. I did not reconcile this.
