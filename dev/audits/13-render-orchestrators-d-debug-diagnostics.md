# Render Orchestrators D (Debug/Diagnostics/Errors) — Deep Audit

Slice: Unit 13 — renderer/render_debug_state_orchestrator.{h,cpp} (1,303 LOC cpp / 153 h), renderer/render_diagnostics_orchestrator.{h,cpp} (1,491 / 64), renderer/rendering_diagnostics.{h,cpp} (170/55), renderer/rendering_error.{h,cpp} (140/80), renderer/render_route_labels.{h,cpp} (352/22). Total ~3,830 LOC.

## Summary

Grade: **C-**.

The code is structurally clean (pimpl-style orchestrators, runtime ports, clear category/severity enums, good route-UID vocabulary) and it mostly tries to do the right observability work. But the "is it useful in production?" answer is **mostly no**: the headline diagnostic features (pipeline trace, splat audit, cull guardrails, state guardrails, anomaly snapshot dump, cross-device op trace, frame-timing ring, production-metrics validation logs) are either **off by default in release** and **off by default in dev**, or are **on but routed to a disabled logger** (LOG_EVERY_FRAMES_VERBOSE == 0). The worst bug is the **singleton race** on `GaussianRenderingDiagnostics::ensure_singleton` (non-atomic double-checked init, called from ctor/dtor/per-frame paths). Second-worst is the **`record_rendering_error` hot-path cost**: unconditional `HashMap[id]++`, `Vector::remove_at(0)` O(n) on a 16-slot ring, context dictionary copy, and a singleton `ensure_singleton` that allocates from the render thread. Third is the **O(n) `remove_at(0)`** pattern used for five separate ring buffers, one of which holds up to 240 samples.

The file write at `dump_pipeline_trace_to_file` runs synchronously inside `update_raster_metrics` on anomaly, and a full `texture_get_data` readback (GPU stall) is piggy-backed on that path. This will absolutely stutter under actual production error conditions — exactly when diagnostics are most needed.

## What this code does

- **`RenderDebugStateOrchestrator`** — owns `DebugConfig` / `DebugState`; reads ~20 project-settings toggles at ctor time; renders HUDs/tile grids/density heatmap by mutating `TileRenderer::RenderParams`; maintains pipeline-trace events; detects anomalies (visible-splat drop, contribution drop, index-mismatch, missing-iterated audit, cull-guardrail drop) and (optionally) dumps a JSON trace + PNG snapshot. Exposes set/get pairs the `GaussianSplatRenderer` binds to GDScript.
- **`RenderDiagnosticsOrchestrator`** — assembles per-frame telemetry snapshots and a "production metrics" subset; validates that subset against a hard-coded contract (`_production_metrics_contract`) and records contract violations; maintains a ring of 240 `FrameTimingSample`s, 64 `CrossDeviceOperation`s, 16 recent errors; exposes `build_render_stats()` for the inspector; manages the error-recovery state machine.
- **`GaussianRenderingDiagnostics`** — process-global singleton coordinating command-line-driven diagnostic runs (`--diagnose-gaussian-rendering`), maintains a cross-renderer recent-error ring (32 slots) and throttled JSON report emission.
- **`RenderingError`** — simple value class, category/severity/code/message/context/recovery; codes memoized via local static `HashMap`.
- **`RenderRouteUID` / `GaussianRenderRouteLabels`** — string-constant vocabulary for render/sort/cull routes and human-readable descriptions for telemetry display.

## DEV_ENABLED / release behavior

This is where the audit bites. Net effect: **almost nothing in this 3,830 LOC slice fires in a release build by default**, and a lot of it doesn't fire in dev either.

- **`DEV_ENABLED` guards in the slice**: exactly **one** — `render_diagnostics_orchestrator.cpp:588` inside `_validate_production_metrics`. `GS_LOG_WARN_DEFAULT` runs always; a bonus `WARN_PRINT` only fires in dev. So in a shipping build a metrics-contract violation goes to the info log only (depending on logger config), never to the editor warning stream. If `GS_LOG_WARN_DEFAULT` is also gated behind a dev log level, release builds are completely silent on contract violations — the whole "validate production metrics" apparatus becomes write-only.
- **`LOG_EVERY_FRAMES_VERBOSE == 0`** (`logger/logging_config.h:6`) hard-disables `_emit_report()` unless `cli_diagnostic_requested` is set. The entire periodic-report path is dead code absent `--diagnose-gaussian-rendering`. The 5s throttle, the 32-slot recent-error ring, and `renderer_reports` assembly are all unreachable in normal runs.
- **`notify_error` is intentionally defanged** (`rendering_diagnostics.cpp:70-73`): the lines that would have armed `cli_diagnostic_requested` and validation layers on error are commented out with "PERFORMANCE FIX: Don't automatically enable diagnostics on error (causes CPU spikes)". That's reasonable damage control but the comment itself is a confession — the diagnostics code path was so expensive it had to be disabled on error. It's now fire-and-forget logging to a ring that nobody reads unless CLI-armed.
- **`_default_pipeline_trace_enabled()`** (`render_debug_state_orchestrator.cpp:32-38`) is hard-coded `false` with a "Disabled for perf baseline" comment. Pipeline trace — the richest observability surface — is off-by-default even in dev. Only enabled by ProjectSettings `enable_pipeline_trace=true` or `enable_all_debug=true`.
- **All ~20 debug toggles** default `false` (`render_debug_state_orchestrator.cpp:321-370`). So `enable_cull_guardrails`, `enable_splat_audit`, `enable_state_guardrails`, `enable_frame_logging`, `enable_binning_counters`, `enable_data_logging` are all dead by default. `_check_cull_guardrails` short-circuits at line 545. `update_raster_metrics` exits at line 645 without populating `splat_audit` or checking anomalies when both trace+audit are off.
- **Anomaly dumps (PNG snapshot + JSON trace to user://) require `enable_data_logging || enable_all_debug`** (`render_debug_state_orchestrator.cpp:727`). Good default (no surprise disk writes), but means the only evidence-capturing path is triple-gated: (1) pipeline trace enabled, (2) a detectable anomaly pattern tripped, (3) data_logging opt-in. In practice nobody in production has this wired up.
- **Cross-device operation trace, texture allocation trace, frame timing history** are all populated unconditionally (no #ifdef guard) and carried for the whole process lifetime inside `DiagnosticsState`. Allocation tax in release is non-zero; observability benefit in release is zero because no one consumes the ring absent `--diagnose-gaussian-rendering`.
- **`record_rendering_error` is unconditional** (`render_diagnostics_orchestrator.cpp:746`). So the error-counting and `ensure_singleton` path runs in every build. See Top issue #1.

**Bottom line**: the module is "off in release, also off in dev" for the expensive stuff, and "always on, never consumed" for the cheap stuff. Worst of both worlds.

## Strengths

- Route-UID vocabulary (`render_debug_state_orchestrator.h:6-51`) is a genuine asset — 25+ named routes with stable string constants, missing-value sentinels, and a normalization helper. The `GaussianRenderRouteLabels` namespace turns those into human prose for the inspector.
- Runtime ports / dependency injection pattern on both orchestrators (`render_debug_state_orchestrator.h:55-67`, `render_diagnostics_orchestrator.h:14-27`) is the right shape for testability, and it's enforced with `ERR_FAIL_COND_MSG` in the ctors (`render_debug_state_orchestrator.cpp:310-317`, `render_diagnostics_orchestrator.cpp:731-743`).
- `_production_metrics_contract()` encodes an explicit telemetry schema as data — key list, expected types, placeholder sentinels. The validator (`render_diagnostics_orchestrator.cpp:428-598`) tags 30+ specific violation types. In principle this is excellent; in practice the warnings go nowhere useful (see release behavior).
- `RenderingErrorCodes::_make_cached` memoizes `Code` constructions — reasonable hygiene, avoids re-allocating `StringName` per call.
- Anomaly detector is principled: `_is_drop(prev, cur, min, divisor)` + cull-guardrail samples keyed by quantized camera pose hash → reproducible per-pose baseline rather than naive frame-over-frame.
- Atomic load of `visible_splat_count` with explicit memory order (`render_debug_state_orchestrator.cpp:642`, `render_diagnostics_orchestrator.cpp:176, 803, 952, 1265, 1450`) — the author understands concurrency at read sites.

## Top issues

1. **[severity: crash]** `renderer/rendering_diagnostics.cpp:17-21` — `ensure_singleton` does `if (!singleton) singleton = memnew(...)` with **no atomic, no mutex, no memory fence**. It is called from `record_rendering_error` (render-thread sorting callbacks: `gaussian_splat_renderer.cpp:934-936`), from `increment_frame_counter` (per-frame, render thread), from `register_renderer` / `unregister_renderer` (scene-director thread), from `register_types.cpp:115` (main thread, module init), and from the dtor's `unregister_renderer` call (`gaussian_splat_renderer.cpp:1151-1152`). If two renderers are constructed concurrently (e.g. async asset load from the loader thread) both can observe `singleton == nullptr` and both `memnew`, leaking one and leaving dangling pointers in one of them. — **Matters because**: this is a process-wide singleton; double-init will lose one renderer's registered-renderers list and corrupt `recent_errors`. — **Fix**: initialize at module-register time only (`register_types.cpp` already calls `ensure_singleton()` there), make all other call sites `if (get_singleton())` without the ensure, and drop the lazy-init branch entirely. Or use `SafeNumeric`/mutex.

2. **[severity: perf]** `render_debug_state_orchestrator.cpp:728-745` — anomaly-dump path does `dump_pipeline_trace_to_file(...)` (JSON stringify + synchronous `FileAccess::WRITE` to `user://`) **and** `device->texture_get_data(final_texture, 0)` (`_save_texture_snapshot`, line 144) **and** `image->save_png(...)` — all inside `update_raster_metrics`, which is called from the per-frame render path. `texture_get_data` forces a GPU→CPU readback; on D3D12/Vulkan that's typically a full pipeline flush (tens of ms). PNG encode on the render thread is another 5–50 ms. — **Matters because**: when diagnostics finally fire (the moment they're most valuable — a real anomaly), they induce a hitch that's indistinguishable from the rendering bug itself, masking the symptom and confusing frame-time analysis. Cooldown is 30 frames so a persistent anomaly dumps ~twice per second. — **Fix**: queue the dump (path + RID + frame_id) into a background task; use `BatchedAsyncReadback` (already in the module) for the texture; write PNG on a worker.

3. **[severity: perf]** `render_diagnostics_orchestrator.cpp:786-790` and **multiple ring buffers** using `Vector::remove_at(0)`: `cross_device_operations` (64), `frame_timing_history` (240, line 807), `recent_errors` (16, line 757), `production_metrics_summaries` (line 704), `sort_metrics_history` (120+, line 1215), `recent_errors` in `GaussianRenderingDiagnostics` (32, `rendering_diagnostics.cpp:167`). `Vector::remove_at(0)` is O(n) memmove. For the 240-slot frame-timing ring, every frame shifts 239 samples down by one. — **Matters because**: this runs unconditionally every frame in release. At 60 Hz × 240 entries × ~40 bytes = ~14 MB/sec of pointless memmove. — **Fix**: use a proper ring buffer (head+tail indices into a fixed-size array) or at minimum a `LocalVector` with `write[0] = back(); resize(size()-1)` trick. Same pattern everywhere.

4. **[severity: maint]** `render_diagnostics_orchestrator.cpp:156-162, 1295` — `_merge_dictionary` iterates `Array keys = p_source.keys()` (allocates) then does `p_target[key] = p_source[key]` per key (variant copy). In `build_render_stats`, this merges `last_telemetry_snapshot` into `stats` — a Dictionary with 100+ keys. `duplicate()` at line 1295 also deep-copies. Both run every frame `finalize_frame_metrics` and every `get_render_stats` call. — **Matters because**: editor inspectors poll `get_render_stats` at ~10 Hz; each call now triggers two 100-key Dictionary copies. Combined with issue #3 it's why the `//PERFORMANCE FIX` comment in `notify_error` exists. — **Fix**: cache by generation counter; if `frame_counter` unchanged, return the cached dict. Or build `stats` once in `finalize_frame_metrics` and just return a reference-copy.

5. **[severity: crash]** `render_debug_state_orchestrator.cpp:544-618` — `_check_cull_guardrails` holds an in-place pointer (`CullGuardrailSample *match`) that can be invalidated by the ternary at line 605: `match ? *match : cull_guardrail_samples[cull_guardrail_cursor++ % kCullGuardrailSamples]`. Actually in this case it's a fixed C array so no invalidation — **but** `cull_guardrail_cursor` is `uint32_t` and increments unbounded; with `kCullGuardrailSamples=64` the modulo is safe, but nothing prevents concurrent access: `update_raster_metrics` may run from the render pipeline while the inspector reads `debug_state` from the main thread. — **Matters because**: the class has no synchronization; a main-thread reader can see torn `CullGuardrailSample` values. Also `cull_guardrail_cursor++` is not atomic. — **Fix**: document that this orchestrator is render-thread-only and add a thread-id assert; or actually lock.

6. **[severity: maint]** `render_debug_state_orchestrator.cpp:415-422` — the ctor does `enable_frame_logging_verbose → turn on sort/tile/pipeline/dispatch/gpu_counter/autotune logs` but also has a sibling branch at 372-392 for `enable_all_debug`. Both branches are fragile to add a new sub-toggle and forget one side. The fallback branch at 393-413 for "no ProjectSettings" duplicates all the defaults. — **Matters because**: three places to keep in sync; drift is guaranteed. The `enable_all_debug` branch forces `frame_log_frequency >= 1` (line 389) but the fallback doesn't. Silent-diverging semantics. — **Fix**: single canonical default struct + one "apply enable_all" function; no fallback branch — set defaults once then conditionally load settings.

7. **[severity: perf]** `render_diagnostics_orchestrator.cpp:1288, 1300-1305` — `_load_production_metrics_config()` parses six project-settings every `finalize_frame_metrics` call (i.e. every frame). ProjectSettings lookups are HashMap lookups on `StringName` and they're not free. Then `_validate_production_metrics` runs the full 30-rule validator every frame. — **Matters because**: you pay the config read cost per-frame even when validation is disabled (the early-exit is on the result, not on the load). And the validator allocates ~10 small `String`s per invocation. — **Fix**: load config once and cache with a ProjectSettings-version stamp; only re-load on change. Move the validator behind `#ifdef DEV_ENABLED` — it's a contract checker, not runtime logic.

8. **[severity: maint]** `render_diagnostics_orchestrator.cpp:27-39, 188-201 (in debug orchestrator)` — `_stage_status_to_string` is duplicated between the two orchestrators. Same enum, same strings, two copies. Similarly `_normalize_route_uid_for_stats` and `_normalize_route_uid_for_snapshot` are near-identical. Five different `_normalize_*` helpers across the two files plus `render_route_labels.cpp`. — **Matters because**: every new route UID requires touching three files. — **Fix**: move to `render_route_labels.{h,cpp}` (already has `GaussianRenderRouteLabels`).

9. **[severity: maint]** `rendering_diagnostics.h:55` — the header has `#include "../logger/logging_config.h"` **after** `#endif // GAUSSIAN_RENDERING_DIAGNOSTICS_H`. Misplaced include — it's outside the include-guard, so every translation unit re-processes it. Currently benign because the file has only constants, but it's a latent ODR/include-order landmine; if someone adds state to `logging_config.h` this breaks. — **Matters because**: the header ends in a syntactic trap that survives review unless someone looks at raw bytes. — **Fix**: move the include above the `#endif`, or better, into the .cpp (it's only used in `rendering_diagnostics.cpp:134`).

10. **[severity: maint]** `rendering_error.cpp:93-102` — `_make_cached` uses a function-local `static HashMap<uint64_t, RenderingError::Code> cache`. Function-local statics have guaranteed thread-safe init in C++11+, but **subsequent writes are not synchronized**. Every call site reads+writes this HashMap under concurrent render-thread calls. — **Matters because**: `device_unavailable()` etc. are called from any thread that can fail (sorting, readback callbacks). Two threads calling `gpu_sort_unavailable()` for the first time concurrently race. In practice the first-populate race is the only window (subsequent calls read-only), but it exists. — **Fix**: populate once at module init or make the codes `constexpr` / `inline constexpr` values; they're literally a (Category, int, const char*) triple — no runtime allocation needed.

11. **[severity: perf]** `render_debug_state_orchestrator.cpp:857-941, 943-945` — `get_pipeline_trace_snapshot()` builds a Dictionary with nested Dictionaries for every pipeline event (can be ~50+ per frame when trace is on), plus stage-io, stage-metrics, stage-results. `dump_pipeline_trace_to_file` calls `get_pipeline_trace_json()` → `JSON::stringify(..., "  ")` with pretty-print indentation. Running this on an anomaly dump adds ~5–20 ms. Called from the **render thread inside `update_raster_metrics`** via issue #2. — **Matters because**: anomaly path is already terrible; this makes it worse. — **Fix**: write minified JSON on the hot path; pretty-print only in explicit user-invoked dumps.

12. **[severity: maint]** `render_diagnostics_orchestrator.cpp:547-556` — the "placeholder timing" validator adds three issues (`cull_ms_placeholder`, `sort_ms_placeholder`, `raster_ms_placeholder`) whenever stage timing is `<= 0.0f` but status is `success`. But many legitimate short stages round to `0.0f` (cull on 512 splats is <50 µs). Threshold is `>= 1024` splats for "meaningful workload" but even 100k splats can cull in <0.01 ms on a modern GPU when the frustum is empty. — **Matters because**: false-positive contract violations drown out real ones; the `production_metrics_invalid_count` counter becomes meaningless. — **Fix**: either remove these placeholder checks or move to `<= 0.0001f`-style epsilon that's genuinely implausible.

13. **[severity: perf]** `render_diagnostics_orchestrator.cpp:949` — `build_render_stats` calls `_merge_dictionary(stats, diagnostics_state.last_telemetry_snapshot)` unconditionally if `last_telemetry_snapshot` is non-empty, overriding all the `else`-branch fallbacks. This means the 100+ key telemetry is always merged into the per-call stats dictionary even when the caller (e.g. a unit test) only wants a 5-key subset. No way to ask for a lightweight view. — **Matters because**: makes the inspector snapshot expensive and couples telemetry-completeness to stats-basic-info. — **Fix**: split into `build_render_stats_minimal()` (5 keys) and `build_render_stats_full()` (current behavior); expose both through the renderer.

14. **[severity: maint]** `render_diagnostics_orchestrator.cpp:754` — `diagnostics_state.runtime_error_statistics.error_code_counts[p_error.get_code().id]++` silently inserts on miss. The code IDs come from `RenderingErrorCodes` namespace but also from any caller that constructs `make_code(category, id, label)` directly — nothing prevents id collisions across categories. — **Matters because**: `error_code_counts` is keyed by int (id) not by (category,id) pair; a `DEVICE:100` and a `PIPELINE:100` would collide in the counter HashMap. Today codes don't collide, but the contract is implicit. — **Fix**: key by `(category << 32) | id` (same scheme `_make_cached` already uses); or better, by `StringName label`.

## Cross-cutting patterns

- **"Off-by-default observability"**: every expensive diagnostic is gated behind a ProjectSettings bool defaulting to `false`. Good for perf; terrible for post-mortem ("can you repro with `enable_pipeline_trace=true`?" — no, because the bug already shipped).
- **Write-only rings**: `frame_timing_history`, `cross_device_operations`, `texture_allocation_trace`, `recent_errors` (×2), `sort_metrics_history` are populated unconditionally but consumed only via `get_runtime_diagnostic_snapshot()` which is only wired into `--diagnose-gaussian-rendering`. In a normal release run they're memory/CPU overhead with zero readers.
- **`Vector::remove_at(0)` as a ring-buffer substitute**: repeated 6+ times. The module clearly needs a `RingBuffer<T, N>` primitive.
- **Callback-ported orchestrators**: both orchestrators take member-function-pointers via `RuntimePorts` to call back into `GaussianSplatRenderer`. This is the right decoupling pattern but it puts a hard dependency on `gaussian_splat_renderer.h` (see includes) anyway, so the theoretical testability gain is unrealized. Could simplify to direct calls.
- **Dictionary-of-everything for telemetry**: every metric is stuffed into a flat `Dictionary` with string keys and auto-converted Variants. ~120 keys per snapshot. No schema type-safety, validator has to re-parse types on read.
- **Thread-unsafe-by-convention**: no locks, no atomics (beyond `visible_splat_count`), comments don't assert which thread owns which state. `ensure_singleton` is the worst offender but the orchestrators are also implicitly render-thread-only with no enforcement.

## Recommended refactor moves

**P0 (do now, ~1–2 days each)**
- Fix `ensure_singleton` race: drop lazy init, require `register_types.cpp` to initialize and tear down; convert all lazy call sites to `if (get_singleton())`. — **~2 hours**.
- Move anomaly PNG + JSON dump off the render thread: enqueue `{frame_id, RID, trace_json_string}` into a `WorkerThreadPool` task. — **~1 day**.
- Replace the 6 `Vector::remove_at(0)` ring buffers with a `FixedRingBuffer<T,N>` template (head+tail+wrap). — **~1 day**.

**P1 (this sprint, ~2–4 days)**
- Gate `_validate_production_metrics` behind `#ifdef DEV_ENABLED` (or a single "diagnostics_validation_enabled" compile flag), cache `_load_production_metrics_config` per ProjectSettings version stamp. — **~4 hours**.
- Dedupe the 3× `_stage_status_to_string` / 5× `_normalize_*_route_uid_*` helpers into `render_route_labels.cpp`. — **~2 hours**.
- Collapse the three-branch default-resolution in `RenderDebugStateOrchestrator` ctor into a single-path "init defaults + conditionally override from settings". — **~4 hours**.
- Split `build_render_stats` into minimal/full variants and call minimal from the inspector hot path. — **~1 day**.
- Fix the `rendering_diagnostics.h` misplaced `#include` after `#endif`. — **~5 minutes**.

**P2 (tech-debt / nice-to-have, ~1 week+)**
- Switch `RenderingError::Code` to `inline constexpr` values, drop `_make_cached`'s HashMap. — **~2 hours**.
- Introduce a typed telemetry struct (POD) populated in `finalize_frame_metrics`; keep the Dictionary projection only at the script-binding boundary. — **~3–4 days**.
- Add thread-safety contracts as `DEV_ASSERT(thread_caller::is_render_thread())` at orchestrator entry points. — **~1 day**.
- Ship a `GS_CI_DIAG_DUMP=1` env-var path that forces the anomaly dump in dev builds without requiring the full `enable_all_debug` cascade. — **~4 hours**.

## Blind spots

- **Out-of-slice**: `DebugOverlaySystem`, `DebugOverlayQueryView`, `DebugOverlayCommandSink`, `TileRenderer`, `GPUCuller`, `PainterlyRenderer`, `OutputCompositor`, `GaussianSplatRenderer::FrameStateProvider`, `GS_LOG_*` macros, `gs_debug_trace.h`, `GaussianEffectiveConfig`. The orchestrators delegate heavily to these and I have not verified their thread-safety, allocation profile, or release behavior. In particular `DebugOverlaySystem::build_command_sink(renderer)` is called multiple times per setter in `RenderDebugStateOrchestrator` — if that call allocates, the setter cost is hidden.
- **GS_LOG_* gating**: the `GS_LOG_WARN_DEFAULT` / `GS_LOG_INFO_DEFAULT` / `GS_LOG_GPU_SORT_INFO` macros determine whether the non-`DEV_ENABLED` log calls actually emit in release. I did not read `logger/gs_logger.h`. If those macros compile to no-ops in release, issue #1 in "DEV_ENABLED / release behavior" becomes worse — zero observability in shipped builds.
- **`GaussianSplatRenderer::DebugConfig` / `DebugState` / `DiagnosticsState` layout**: defined in `gaussian_splat_renderer.h` (not in slice). I assumed these structs match how the orchestrators use them; silent field-type drift is possible.
- **Whether `update_raster_metrics` runs per-frame or only on a code path**: I traced it via `GaussianSplatRenderer::update_debug_raster_metrics` but did not follow every caller in `render_pipeline_stages.cpp`. Worst-case is per-frame, which is how I rated issue #2; if it's actually once-per-anomaly-already-rate-limited the severity drops.
- **`--diagnose-gaussian-rendering` CI usage**: the `GS_LOG_INFO_DEFAULT("[GaussianDiagnostics][CI] ...")` tag suggests CI consumes this, but I did not inspect CI scripts.
- **`production_metrics_summaries` persistence**: stored in `DiagnosticsState` in-memory only; no serialization to disk. If the intent is long-run perf tracking, there's a missing sink. I did not look for external consumers outside the slice.
