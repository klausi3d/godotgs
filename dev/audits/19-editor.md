# Editor Integration ‚Äî Deep Audit

## Summary

**Grade: C+**

The editor integration is *functionally broad* ‚Äî gizmo, inspector, import dialog, resource preview generator, hot-reload, 3D preview control, brush tooling, and advanced import-settings dialog all exist. Architecture is conventional for a Godot editor plugin. `TOOLS_ENABLED` gating is consistent across every file (verified in all 11 headers/bodies).

But the implementation has real hazards. The earlier audit findings are **confirmed**: thumbnail generation runs synchronously on the main thread, pre-import validation is non-existent in the "Open File" entry point (we cheerfully try to `load_file` huge PLYs in a modal dialog), and there is no GPU-readback timeout or bound anywhere. Additionally I found: a thumbnail pixel loop using `Image::set_pixel` for up to 262k pixels per call (O(N) with per-call Variant boxing), a lambda-captures-by-ref pattern in `_process_hot_reload_for_watch` that is legal but trips on reentrancy, a `SubViewport::UPDATE_ALWAYS` live 3D preview embedded in the inspector for every Gaussian asset (renders even when the inspector scrolls off-screen), and a `EditorFileDialog` mode/access mismatch between `_initialize_editor_ui_once` (ACCESS_FILESYSTEM) and `GaussianEditorIntegration::_ensure_file_dialog_filters` (ACCESS_RESOURCES) ‚Äî the second call silently wins because of its `filters_configured` guard, but anyone reading the code sees contradictions.

Cache eviction is LRU-*ish* but uses `O(n)` list removal; hot-reload watch removal has an O(n¬≤) de-duplication loop; `_compute_asset_fingerprint` returns all the packed arrays by value (CoW makes this cheap in Godot, fine in practice). Brush undo/redo correctness is decent. The `GaussianImportDialog` declares three methods (`_schedule_preview_update`, `_on_preview_debounce_timeout`, `_process_queued_preview`) that are **never implemented** ‚Äî dead declarations in the header.

There is no automatic teardown of the `gaussian_import_settings_dialog` in `NOTIFICATION_EXIT_TREE`, but since it's a child Node the SceneTree handles it. Inspector plugin re-parse fires every selection; brush-session state is per-plugin-instance, and the plugin is a long-lived Ref ‚Äî state survives (intentional per hint text), but is silently retained across completely different nodes.

Overall: it mostly works, performance is acceptable for small splats, but the main-thread risks on large imports are real and were called out earlier and remain **unmitigated**.

## What this code does

1. **`GaussianEditorPlugin`** (`gaussian_editor_plugin.{h,cpp}`) ‚Äî Main entry point. Registered via `EditorPlugins::add_by_type<>` at `MODULE_INITIALIZATION_LEVEL_EDITOR`. Owns:
   - An `EditorFileDialog` for "Import Gaussian splat‚Ä¶" flow.
   - A `GaussianImportDialog` (simpler tabbed dialog) and a `GaussianImportSettingsDialog` (FBX-style advanced dialog, opened from .ply/.spz double-click).
   - A `GaussianSplatGizmoPlugin`.
   - Four `EditorInspectorPlugin` subclasses (data, renderer, asset, node).
   - A `GaussianSplatAssetPreviewGenerator` registered with the resource previewer.
   - A runtime `GaussianThumbnailGenerator`.
   - A polling hot-reload subsystem keyed by source path / project path.
   - Inspector stats bindings (Label pointers updated every 15 frames from `NOTIFICATION_PROCESS`).
2. **`GaussianEditorIntegration`** ‚Äî Refcounted helper for path normalization and filesystem-changed callbacks. Sets filters on the EditorFileDialog.
3. **`GaussianEditorServices`** ‚Äî Free-function namespace: dict accessors, stat/metadata formatting, error‚Üíhuman-string translation.
4. **`GaussianSplatGizmoPlugin`** ‚Äî Standard `EditorNode3DGizmoPlugin`: bounds box, LOD circles, stats cross, performance bars, and a per-splat preview (wireframe/points/heatmap) up to 1000 sample points.
5. **`GaussianThumbnailGenerator`** ‚Äî Pure-software thumbnail rasterizer (color/density/normals/heatmap). In-memory LRU + PNG disk cache under `res://.godot/cache/gaussian_thumbnails`.
6. **`GaussianSplatAssetPreviewGenerator`** ‚Äî `EditorResourcePreviewGenerator` that calls the thumbnail generator.
7. **`GaussianAssetPreviewControl`** ‚Äî A VBoxContainer hosting a `SubViewport` with a live orbit camera, bounds wireframe, and a sampled-down MultiMesh preview (‚â§512 instances). Drag-to-orbit, scroll-to-zoom. Embedded as a custom control in the asset inspector.
8. **`GaussianInspectorPlugins`** ‚Äî Four inspector plugins: asset (preview + reimport), data (stub), renderer (preset buttons + stats), node (large control with quality presets, debug toggles, preview-mode dropdown, brush tools, color-grading bake/restore).
9. **`GaussianImportDialog`** ‚Äî Tabbed AcceptDialog: Quality / Compression / Preview / Metadata. Preview computed by actually running the loader synchronously on the main thread.
10. **`GaussianImportSettingsDialog`** ‚Äî ConfirmationDialog mimicking `SceneImportSettingsDialog`: left 3D viewport with lights, right `EditorInspector` bound to a `GaussianImportSettingsData` (Object with `_get_property_list`).

## Strengths

- Every editor-only compilation unit is wrapped in `#ifdef TOOLS_ENABLED ‚Ä¶ #endif` ‚Äî verified in all 22 files (11 .h + 11 .cpp). No stray editor symbols should leak into export templates.
- `GaussianEditorPlugin::NOTIFICATION_EXIT_TREE` properly removes all four inspector plugins, the gizmo plugin, and the resource preview generator. This matches Godot's "symmetric add/remove in ENTER/EXIT_TREE" pattern.
- Lazy UI construction in `_initialize_editor_ui_once` ‚Äî the ctor explicitly avoids `memnew(EditorFileDialog)` because that calls into `EditorPaths`/`EditorSettings`, which are unavailable under `--headless --test` (excellent defensive note at `gaussian_editor_plugin.cpp:166-170`).
- Disk thumbnail cache with content-derived fingerprint: `_compute_asset_fingerprint` hashes `splat_count`, compression flags, source path, and sampled values from each packed array. Thumbnails survive editor restarts.
- Brush stroke undo/redo via `EditorUndoRedoManager` with a captured "brush-affected state" snapshot (`gaussian_editor_plugin.cpp:594-611`).
- Hot-reload subsystem coalesces poll timers, debounces via a single `SceneTreeTimer` and only schedules when watches exist (`_schedule_hot_reload_poll` at `gaussian_editor_plugin.cpp:697-723`).
- The `GaussianImportSettingsDialog` clears its viewport scene on `NOTIFICATION_VISIBILITY_CHANGED` when hidden ‚Äî bounds/splat node freed eagerly instead of lingering (`gaussian_import_settings_dialog.cpp:212-220`).
- Format-aware UI: `_update_format_specific_controls` hides PLY-only validation options when loading SPZ (`gaussian_import_dialog.cpp:397-435`).
- Legacy-key migration for `.import` sidecars: `GaussianImportSettingsData` has a `legacy_key_map` remapping `asset_type` ‚Üí `general/asset_type` so old projects don't silently lose settings (`gaussian_import_settings_dialog.cpp:468-484`).
- Inspector stats polling is rate-limited: 1 update every 15 `_process` frames (`gaussian_editor_plugin.cpp:304-309`), not every frame.

## Top issues

**[perf]** `editor/gaussian_thumbnail_generator.cpp:324, 350, 429, 459` ‚Äî Every thumbnail generator calls `image->set_pixel(x, y, c)` inside a pixel loop (up to 512√ó512 = 262,144 iterations). `set_pixel` takes a `Color` (4 float Variant boxing), validates the format, does byte-unpack/pack per call. This is the hot path triggered from the asset inspector, the editor resource previewer, the import dialog preview tab, and the fallback texture inside the asset preview control ‚Äî all on the main thread. At size 256 the single ImportDialog preview can stutter for 10‚Äì30 ms on a plain desktop. **Fix:** `image->lock()` + `PackedByteArray` write via `image->get_data()` or use `Image::create_from_data` with a locally filled byte buffer.

**[perf]** `editor/gaussian_thumbnail_generator.cpp:465-534` ‚Äî `generate_thumbnail` is synchronous. The earlier audit flagged this and it is unchanged. There is no async path, no `WorkerThreadPool::add_task`, no cancellation token. For a 10 M-splat asset, `_project_to_canvas` iterates every splat (line 273) on the main thread, which *will* freeze the editor for multiple seconds the first time an asset is inspected. **Fix:** move generation into `WorkerThreadPool::add_task` with a `callable_mp(self, &set_texture).call_deferred` when complete; keep a placeholder texture visible in the meantime.

**[perf]** `editor/gaussian_inspector_plugins.cpp:95-98` and `editor/gaussian_asset_preview_control.cpp:59-63` ‚Äî The asset inspector embeds a `SubViewport` set to `UPDATE_ALWAYS` with `clear_mode=CLEAR_MODE_ALWAYS`. This viewport is created *in the inspector* every time a GaussianSplatAsset is selected. It keeps rendering even if the Inspector panel is scrolled off-screen or the editor window loses focus. For any dataset with a preview, this is a permanent GPU cost until selection changes. **Fix:** switch to `UPDATE_WHEN_VISIBLE` (respects viewport visibility) and only promote to `UPDATE_ALWAYS` when actively drag-orbiting.

**[perf]** `editor/gaussian_import_dialog.cpp:329-395` (`_load_preview_asset`) ‚Äî The simpler import dialog synchronously instantiates `PLYLoader`/`SPZLoader` and calls `load_file(source_path)` on the main thread. There is no size check, no byte-count guardrail, no progress UI, no cancel. On a 5 GB `.ply` this hangs the editor for minutes. **Fix:** stat the file first; if over a threshold (e.g., 200 MB), either reject with an error or move the load to a worker; or sniff just the PLY header for stats without reading the body for preview.

**[perf]** `editor/gaussian_import_dialog.cpp:726-752` (`_update_preview`) ‚Äî Called from every `_on_settings_changed` which fires on every single checkbox toggle, spinbox tick, preset change (`_refresh_all` ‚Üí `_update_preview`). Each call re-generates a thumbnail synchronously. The dialog declares `preview_debounce_timer`, `preview_request_serial`, `preview_update_pending`, `preview_generation_in_progress`, and three helper methods (`_schedule_preview_update`, `_on_preview_debounce_timeout`, `_process_queued_preview`) ‚Äî **all declared in the header, none defined in the cpp**. Dead infrastructure that was meant to solve exactly this problem. **Fix:** either implement the debounce (the state is there) or delete the dead declarations.

**[crash]** `editor/gaussian_editor_plugin.cpp:768-782` (`restore_context` lambda in `_process_hot_reload_for_watch`) ‚Äî Captures `[&]` with references to `previous_node`, `previous_renderer`, `previous_asset`, `previous_source_path`, `previous_import_options`, and implicitly `this`. The lambda is invoked after potentially calling `_import_from_path` which may itself `active_asset = refreshed_asset`, free the previous asset (dropping the Ref), and recurse into `_refresh_active_asset_metadata(true)` ‚Üí `_sync_ui_from_asset` ‚Üí `_track_hot_reload_source` ‚Üí back to `_schedule_hot_reload_poll`. The `previous_*` locals are held by value in the stack frame of `_process_hot_reload_for_watch`, so the lambda is safe only if no inner call shreds the stack. In practice nothing frees stack locals, but because `previous_node` is a raw pointer it can dangle if the node is queue_freed from an inner callback. **Fix:** replace `previous_node` with `ObjectID` and re-resolve via `ObjectDB::get_instance` before dereferencing.

**[corruption]** `editor/gaussian_editor_plugin.cpp:177-191` vs. `editor/gaussian_editor_integration.cpp:38-48` ‚Äî Two places configure the same `EditorFileDialog`. `_initialize_editor_ui_once` sets `FILE_MODE_OPEN_FILE` + `ACCESS_FILESYSTEM` *before* `add_child`. Then on `NOTIFICATION_ENTER_TREE` we call `editor_integration->setup(this, import_dialog)`, which calls `_ensure_file_dialog_filters`, which **overwrites** access with `ACCESS_RESOURCES` and also clobbers filters. The `filters_configured` guard means this only happens once, but the final state is inconsistent with the first call's intent (which was `ACCESS_FILESYSTEM` ‚Äî i.e., allow arbitrary paths, then reject non-`res://` in `_on_import_file_selected`). With `ACCESS_RESOURCES` the user literally cannot browse to a path outside the project ‚Äî the `show_warning("must be inside the project")` branch in `_on_import_file_selected` is now unreachable, which is silent behavior drift. **Fix:** pick one access mode and delete the other.

**[maint]** `editor/gaussian_editor_plugin.cpp:313-335` (`edit`) ‚Äî Never clears old inspector-stats bindings, so if the user selects A, then B, then A again, both Labels get added to `inspector_stats_bindings` (the old Label is pruned only when `tree_exiting` fires, which happens when the inspector rebuilds). Reasonable, but the binding list grows unboundedly across selections because `parse_begin` always creates a *new* Label and appends a new binding. The only pruning happens via a tree_exiting signal wired from `register_inspector_stats` ‚Äî if the Label outlives the inspector (it shouldn't, but if the editor holds it for snapshotting), this leaks. Low risk but worth a sweep: ensure `tree_exiting` really fires in all cases.

**[maint]** `editor/gaussian_editor_plugin.cpp:473` ‚Äî `last_import_options.clear()` in `_clear_selection` erases user intent. Next time the user opens the same asset and the metadata lookup misses (e.g., the cached Dictionary was nulled), we fall through to `effective_options = last_import_options` with an empty Dictionary, and `_dictionary_to_hashmap` returns empty parameters. The importer then applies defaults. This can silently change user-selected options on repeat imports from different asset sources. **Fix:** don't `.clear()` `last_import_options` in `_clear_selection`; scope it to actual fresh sessions.

**[perf]** `editor/gaussian_thumbnail_generator.cpp:150-166` ‚Äî `_touch_cache_key` does `O(n)` linear scan + `remove_at(0)` to prune oldest; `_prune_cache_if_needed` does `remove_at(0)` in a loop until size ‚â§ 64. With 64 entries this is fine, but the combined shape is a non-canonical LRU that re-walks the vector on every single touch, and `_thumbnail_cache_order` is a `Vector<String>` (copies on every resize). **Fix:** use a doubly-linked `List<String>` with `HashMap<String, List<String>::Element*>` for O(1) LRU.

**[perf]** `editor/gaussian_editor_plugin.cpp:109-121` (`_append_unique_hot_reload_node_id`) and `:123-154` (`_collect_live_hot_reload_nodes`) ‚Äî Both contain O(n¬≤) de-dup loops over `Vector<ObjectID>`. With a handful of watched nodes this is cheap; if a scene has 50 gaussian nodes reloading simultaneously (`_on_hot_reload_timer_timeout` iterates every watch and calls the de-dup per watch) it's O(watches ¬∑ nodes¬≤). **Fix:** wrap the vector in a `HashSet<ObjectID>` or guard with size budget.

**[maint]** `editor/gaussian_splat_gizmo_plugin.cpp:197-227` (`draw_lod_radius`) ‚Äî Comment says "Draw circles at different LOD distances", but the actual `max_distance` is sourced from `p_node->get_max_render_distance()`, then the three LODs are hardcoded fractions 25/50/100 %. That may or may not match the runtime LOD selector's thresholds (I can't verify ‚Äî renderer is out of slice). If the runtime LOD actually uses different thresholds, the gizmo lies visually. **Fix:** read actual LOD thresholds from the node/renderer, or remove the gizmo and document that it's a rough indicator.

**[maint]** `editor/gaussian_import_dialog.h:151-153` ‚Äî Declares `_schedule_preview_update`, `_on_preview_debounce_timeout`, `_process_queued_preview` that have **zero implementations**. Verified via grep (`Grep "_on_preview_debounce_timeout"` in the cpp returns no matches). If any future code tries to call them, link error; if not, dead declarations confusing future maintainers. **Fix:** delete, or implement the debounce they were meant for.

**[maint]** `editor/gaussian_inspector_plugins.cpp:200-208` ‚Äî `GaussianRendererInspectorPlugin::parse_begin` calls `stats["visible_splats"]` without checking if the key exists. If `get_render_stats()` returns an empty Dictionary (common pre-frame-1), the `int itos(Variant)` path produces `0`, but `String::num(stats["sort_time_ms"], 1)` on a `Variant::NIL` will print `"nan"` or `"0.0"` depending on the Godot version. Minor cosmetic bug, but it's stale data versus the richer `format_gaussian_splat_stats` used for nodes. **Fix:** use `dict_get_*` helpers from `GaussianEditorServices`.

**[maint]** `editor/gaussian_inspector_plugins.cpp:660-796` (brush section) and `:798-844` (color grading section) ‚Äî Both branches enable controls based on `renderer->get_gaussian_data()->get_count() > 0`. Renderer + data may not be populated at inspector-parse time (streaming/lazy load). The user will see brush/grading controls vanish/appear nondeterministically depending on timing. **Fix:** gate on `asset->get_splat_count() > 0` instead (stable resource-time property), or show the section in a "disabled" state with a tooltip.

**[maint]** `editor/gaussian_inspector_plugins.cpp:55-68` ‚Äî `is_renderer_shared_with_other_content` casts to `GaussianSplatSceneDirector::get_singleton()`. If the singleton is null (headless tests, early editor boot), the function returns false and shared-renderer debug toggles become editable ‚Äî but operating on a shared renderer may corrupt the world-submission state. Correct for non-headless but relies on the singleton being ready at inspector-parse time. Worth a `DEV_CHECK` assertion for parity with runtime expectations.

**[maint]** `editor/gaussian_editor_plugin.cpp:673-723` (hot-reload scheduling) ‚Äî No upper bound on `hot_reload_watches` size. If a user opens 500 assets in succession and never closes the plugin, we accumulate 500 watched paths, each polled every 1 second, each doing a `FileAccess::get_modified_time` syscall. Fine for a handful, potential issue with long editor sessions on large projects. **Fix:** cap to N watches with oldest-first eviction.

**[perf]** `editor/gaussian_asset_preview_control.cpp:151-153` ‚Äî In `_update_fallback_texture`, if `asset->get_thumbnail()` returns null we immediately call `thumbnail_generator->generate_thumbnail` synchronously. This happens inside `set_asset` ‚Üí `_rebuild_scene_from_asset` ‚Üí `_update_fallback_texture`, which runs on the main thread during inspector parsing. Same root cause as the other synchronous thumbnail calls. **Fix:** asynchronize or use the already-stored thumbnail when the asset has one.

## Cross-cutting patterns

- **Consistent `TOOLS_ENABLED` gating.** All 22 files bracket bodies with `#ifdef TOOLS_ENABLED ‚Ä¶ #endif` and declarations with the same guard in headers. This is textbook correct; export template builds will exclude editor code.
- **Unregistered `GDCLASS` types.** Several classes use `GDCLASS` (`GaussianThumbnailGenerator`, `GaussianEditorIntegration`, `GaussianSplatGizmoPlugin`, `GaussianSplatAssetPreviewGenerator`, `GaussianAssetPreviewControl`, `GaussianDataInspectorPlugin`, `GaussianRendererInspectorPlugin`, `GaussianAssetInspectorPlugin`, `GaussianSplatNodeInspectorPlugin`, `GaussianImportDialog`, `GaussianImportSettingsDialog`) but only `GaussianEditorPlugin` is `GDREGISTER_CLASS`'d in `register_types.cpp:164`. For internal editor-only types this is mostly harmless (they're not script-visible), but `_bind_methods` in `GaussianThumbnailGenerator` binds `get_cache_statistics` / `clear_cache` etc. that are unreachable. Either register the class or drop the bindings.
- **Raw pointers to editor nodes stored across awaits.** The hot-reload path, the inspector stats bindings, and the brush session all mix `ObjectID` (correct) with raw `Label*`/`Node*` (fragile). The mix is almost right ‚Äî most stale-pointer risks are handled via `ObjectDB::get_instance(node_id)` ‚Äî but `InspectorStatsBinding::label` is a raw `Label*` with only a `tree_exiting` signal guard. If the signal is ever missed (e.g., editor shutdown reordering), `_update_inspector_stats` touches freed memory.
- **Synchronous main-thread work.** Loading preview asset, generating thumbnails, projecting every splat to canvas, building MultiMesh previews ‚Äî all happen inline under `_notification`, `parse_begin`, `set_asset`, `_on_settings_changed`. There is zero `call_deferred` or worker thread usage in this slice. For any non-trivial dataset this is a frozen editor.
- **Comment-vs-code drift.** Multiple inspector methods have multi-line comment plans ("This would add UI for:\n- Data statistics\n- Import/export options\n‚Ä¶") followed by `// For now, just add a simple label`. The code shows a lot of aspirational surface area that isn't implemented. Documentation for future maintainers should prune or realize these.
- **Thumbnail-size clamp is inconsistent.** `gaussian_editor_plugin.cpp:562` clamps 32-512; `gaussian_thumbnail_generator.cpp:467` just asserts `p_size > 0`; `gaussian_resource_preview_generator.cpp:38` takes `MAX(64, ‚Ä¶)` of the incoming size with no upper bound. The previewer could request a 2048-pixel thumbnail and the generator would allocate a 2048√ó2048√ó4-byte = 16 MB image. **Fix:** clamp centrally in `GaussianThumbnailGenerator::generate_thumbnail`.

## Recommended refactor moves

### P0 (ship blockers / correctness & UX)

- **Async thumbnail generation.** Move `generate_thumbnail` to `WorkerThreadPool::add_task`. Return a placeholder immediately; main thread receives the final texture via `callable_mp(self, &on_thumbnail_ready).call_deferred`. Effort: ~1 day. Unblocks the main-thread freeze described in the earlier audit *and* fixes the current `_update_preview` / `_update_fallback_texture` stalls.
- **Pre-import file validation.** Before `_load_preview_asset` or `_load_source_asset`, stat the file: size, extension, header magic for SPZ. Reject anything over a configurable threshold (project setting default: 500 MB) with a friendly dialog. Effort: ~half day.
- **Timebox heavy loader calls.** Add a `std::atomic<bool> cancel` to `PLYLoader::load_file` / `SPZLoader::load_file` (cross-module touch ‚Äî negotiate with owner of `io/`), wire from the import dialog's "Cancel" button. Effort: ~half day + out-of-slice coordination.
- **Fix EditorFileDialog access-mode contradiction.** Either drop `_ensure_file_dialog_filters` (already do the work in `_initialize_editor_ui_once`) or let it own the full configuration. Delete the duplication. Effort: ~15 minutes.
- **Delete dead declarations.** `GaussianImportDialog::_schedule_preview_update`, `_on_preview_debounce_timeout`, `_process_queued_preview` and their related state (`preview_debounce_timer`, `preview_request_serial`, `preview_update_pending`, `preview_generation_in_progress`) are declared but never defined. Either implement or remove. Effort: ~1 hour.

### P1 (correctness & performance)

- **Use a real LRU.** Replace `Vector<String> thumbnail_cache_order` with `List<String>` + `HashMap<String, List<String>::Element*>`. O(1) touch. Effort: ~2 hours.
- **Replace `Image::set_pixel` loops with byte-buffer writes.** Use `Image::create_from_data` with a prefilled `PackedByteArray`. Effort: ~half day. Expected ~10‚Äì30√ó speedup on large thumbnails.
- **Subviewport visibility gating.** Switch `GaussianAssetPreviewControl::viewport->set_update_mode(UPDATE_WHEN_VISIBLE)` and only flip to `UPDATE_ALWAYS` during `orbit_dragging`. Effort: ~1 hour.
- **Harden `_process_hot_reload_for_watch`.** Replace `previous_node` raw pointer with `ObjectID`; re-resolve before dereference in the restore lambda. Effort: ~1 hour.
- **Centralize thumbnail-size clamp.** Clamp `[32, 512]` inside `GaussianThumbnailGenerator::generate_thumbnail` itself. Effort: ~15 minutes.
- **Clamp watched-path set.** `hot_reload_watches.size() > N ‚Üí evict oldest`. Effort: ~1 hour.

### P2 (hygiene)

- **Register or de-bind helper classes.** Either `GDREGISTER_CLASS` the classes whose `_bind_methods` binds public API, or strip the bindings. Effort: ~2 hours.
- **Consolidate inspector stats text.** `GaussianRendererInspectorPlugin::parse_begin` manually formats render stats; `GaussianEditorServices::format_gaussian_splat_stats` does a much richer job. Reuse. Effort: ~1 hour.
- **Document brush-session lifetime.** The paint-inspector explicitly warns users that brush values don't persist, but the session survives selection changes. Either clear on `edit(null)` or re-document. Effort: ~30 minutes.
- **Unit test for hot-reload de-dup.** The O(n¬≤) loops survive because data is small. If they grow, add a regression test. Effort: ~2 hours.

## Blind spots

- **Runtime LOD thresholds.** `draw_lod_radius` uses hardcoded 25/50/100% fractions. I cannot confirm they match the actual `GaussianSplatRenderer` LOD selector without touching out-of-slice renderer code.
- **Editor singletons in headless tests.** The ctor comment references past crashes in `gaussian_production_gates`. I did not verify the test harness still reproduces the pre-fix crash if ctor-time construction were restored. The defensive pattern looks correct; I trust the comment.
- **GPU-readback timeout.** The earlier audit flagged "no GPU-readback timeout." In this slice there is **no GPU readback at all** ‚Äî thumbnails are software-rasterized on CPU. The flag likely belongs to a different slice (`renderer/` or the actual runtime thumbnail generator that uses `RenderingServer`). If it does exist in this slice, I didn't see it.
- **`GaussianImportSettingsData` lifetime under re-open.** The ctor `memnew`s `settings_data` once; `_populate_settings_data` clears + repopulates `current/defaults`. I did not trace whether `inspector->edit(nullptr)` in `~GaussianImportSettingsDialog` runs before `memdelete(settings_data)` ‚Äî it does in source order, but Godot node destruction order may call dtor twice if dialog is a child node. Potential double-free not confirmed.
- **`GaussianSplatNode3D` method signatures.** I read the editor code; I did not verify every `set_*`/`is_*` signature compiles against the actual `GaussianSplatNode3D` header. The audit assumes `is_preview_enabled`, `is_showing_bounds`, `is_showing_residency_hud`, `get_debug_draw_mode`, `set_splat_asset`, `is_painterly_enabled`, `is_color_grading_baked`, `bake_color_grading_snapshot`, `restore_color_grading`, etc. all exist. The inspector plugin breaks hard if any of these rename.
- **Concurrency on `HotReloadWatch::node_ids`.** The `SceneTreeTimer` callback runs on the main thread, so no race. But `_test_process_hot_reload_path_now` is a test helper that could be called from anywhere. If tests ever run it off the main thread, data races on `hot_reload_watches`. Unlikely but unverified.
- **`EditorUndoRedoManager` interaction with the director.** `_commit_runtime_modifications` / `_revert_runtime_modifications` both call `splat_data->commit_runtime_changes()` ‚Äî I did not verify these methods interact correctly with the scene director's submission tracking.
- **Fallback texture memory leak in `_update_fallback_texture`.** Each call generates a fresh `Texture2D` (if not stored) and hands it to `TextureRect::set_texture`. The old texture is dropped by Ref semantics. This looks correct; I did not valgrind-verify under repeated selection.
